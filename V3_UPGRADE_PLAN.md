# Boom V3.0 升级规划方案 (Draft)

本项目旨在将 Boom V2.5 升级为 V3.0 大版本。V3.0 的核心目标是在坚持“无害、可控”的前提下，大幅提升**视觉震撼度**、**交互趣味性**以及**远控端的控制体验**。

以下是初步的功能构想清单，供讨论与后续实施参考。

*   **视觉调整**：保持 V2.5 原有的清爽复古面板风格，不搞花里胡哨，但进行全面、彻底的中文化翻译。
*   **交互优化**：重新梳理操作面板的逻辑结构。将众多整蛊功能（原先堆砌在一个大格子里）按“视觉弹窗”、“物理/输入干扰”、“破坏性伪装”、“系统漏洞模拟”等合理维度进行分类（可采用卡片折叠或 Tab 标签页），确保控制台指令清晰易懂，操作绝不混乱。

*   **免杀考量**：在打包 `prank_release.py` 时优化混淆策略，降低被安全软件误报拦截的几率。

## 十一、 通信安全与防劫持 (Communication Security)

> ⚠️ **V2.5 已知安全弱点速览**：
> - Token 在 `index.html` JS 中明文传输，任何能打开控制台的人都能截取
> - `is_controller` 的判断只靠 `client_id` 字符串前缀 (`controller_` / `web_`)，知道规则就能伪造身份
> - 使用明文 `ws://`，局域网内抓包即可拿到 Token 和所有指令内容
> - 无速率限制，可被暴力穷举 Token
> - 客户端对收到的指令无签名验证，中间人可替换指令内容

### 1. 全链路 TLS 加密 (WSS Upgrade)
- **问题现状**：明文 `ws://`，局域网内抓包即可截取 Token 和所有指令。
- **升级方案**：服务端升级为 `wss://`，配置自签证书（内网）或 Let's Encrypt（公网）。`build_release.py` 同步支持一键生成带 TLS 指纹的客户端。

### 2. TOTP 双因子认证 (Two-Factor Auth)
- **问题现状**：静态 Token 一旦泄漏，控制权永久丢失。
- **升级方案**：控制端登录同时需要 **静态 Token** + **TOTP 动态码**（每 30 秒更新，类 Google Authenticator）。即使 Token 被偷，没有动态码依然进不来。
- **依赖库**：服务端 `pyotp`，网页端用标准 TOTP 算法实现。

### 3. 指令 HMAC 签名校验 (Message Signing)
- **问题现状**：客户端盲目执行任何格式合法的 JSON 指令，中间人可在服务器和客户端之间篡改指令内容。
- **升级方案**：服务端转发指令时附加 `HMAC-SHA256` 签名（用共享密钥生成）。客户端每次执行前先验签，篡改或伪造的指令直接丢弃。

### 4. 控制端 IP 白名单 (Controller IP Whitelist)
- **问题现状**：`is_controller` 的判断逻辑只靠 client_id 前缀，任何人只要构造 `web_xxx` 的 ID 就能拿到控制权。
- **升级方案**：服务端配置文件中预设允许接入控制端的 IP 段（如自己的手机 IP / VPN 出口 IP），来源不在白名单的连接一律以客户端身份处理，无法下发指令。

### 5. 暴力穷举防护 (Rate Limit & Lockout)
- **问题现状**：无频率限制，可无限尝试 Token。
- **升级方案**：
  - 单 IP 连续 5 次 Token 验证失败后，自动封禁 10 分钟。
  - 短时间内大量 WebSocket 连接请求（如 > 20 次/秒）直接封禁该 IP。
  - 封禁事件实时推送给控制端，告警提示。

### 6. 会话绑定与一次性 Token (Session Binding)
- **问题现状**：Token 可以被重放，同一个 Token 在任意时间任意地点都有效。
- **升级方案**：控制端登录成功后服务端颁发一个**带过期时间的 Session Token**（如有效期 8 小时），后续所有指令使用该 Session Token 而非原始 Token 通信，原始 Token 不在网络上重复传输。

## 十二、 性能与架构优化 (Performance & Architecture)

### 1. 智能心跳与实时掉线检测 (Smart Heartbeat)
- **问题现状**：客户端固定每 10 秒发一次心跳，服务端被动等待，无法主动检测断线。控制端要等到下一次 `list_clients` 请求才能发现客户端已经离线。
- **升级方案**：服务端主动对所有客户端发送 Ping，超过 30 秒无响应即标记为"已断线"并实时推送给控制端，而非依赖轮询。

### 2. 断线指数退避重连 (Auto-Reconnect with Backoff)
- **问题现状**：`prank_release.py` 断线后不会重连，整蛊链路彻底中断。
- **升级方案**：实现指数退避重连策略：断线后等待 2s 重试 → 失败则 4s → 8s → 最长不超过 120s，直到重连成功。重连期间客户端保持自身状态，重连后自动上报当前 Stage。

### 3. 操作审计日志 (Audit Log)
- **功能描述**：服务端记录所有控制端登录事件、每一条下发指令、目标客户端 ID、时间戳，写入带自动轮转的本地日志文件。出了纠纷时可追溯到底。

### 4. 客户端身份指纹绑定 (Client Fingerprinting)
- **功能描述**：客户端首次连接时，服务端基于 `hostname + 网卡 MAC + 硬盘序列号` 生成唯一指纹并存档。后续每次连接携带该指纹，防止打包后的 exe 被复制到无关机器上滥用。

### 5. 统一配置文件 (Config File)
- **问题现状**：Token、端口、服务器地址全部散落在源码硬编码，修改需要改多处文件。
- **升级方案**：引入 `boom.conf.toml` 统一管理所有配置项，首次启动向导式交互生成，后续服务端和打包工具均从同一配置文件读取，杜绝漏改。

### 2. 窗口无限繁殖 (Infinite Window Spawn)
- **表现形式**：每当受害者关闭一个整蛊弹窗，立刻弹出两个同样的窗口来"顶替"——像打地鼠一样，关得越快，弹得越多，直到屏幕被塞满。
